Abstract

The Mission vocabulary defines concepts and properties for describing missions in terms of their objectives, their constituent components, the functions those components perform, and the requirements that specify them.

3. Aspects

3.1. PerformingElement

PerformingElement is the domain of performs. Any concept that performs a Function should specialize PerformingElement.

Subtypes
  • Component
  • Mission
Relations with source
  • PerformingElement ← Performs → Function
  • PerformingElement ← Produces → Product

3.2. PresentingElement

PresentingElement is the domain of presents. Any concept that presents an Interface should specialize PresentingElement.

Subtypes
  • Component
  • fse:EndCircuitPresentingElement
  • Interface
  • Mission
Relations with source
  • PresentingElement ← Presents → Interface

3.3. SpecifiedElement

SpecifiedElement is the range of specifies. Any concept or reified relation that isSpecifiedBy a Requirement should specialize SpecifiedElement.

Subtypes
  • analysis:Characterizes
  • Component
  • base:Contains
  • Mission
  • Performs
  • Presents
  • Produces
  • TransfersIn
  • TransfersOut
Relations with target
  • Requirement ← Specifies → SpecifiedElement

3.4. TraversingElement

TraversingElement is the domain of traverses. Any concept that traverses an Interface should specialize TraversingElement.

Subtypes
  • Flow
  • Item
Relations with source
  • TraversingElement ← Traverses → Junction
Relations with target
  • Interface ← Transfers → TraversingElement
  • Interface ← TransfersIn → TraversingElement
  • Interface ← TransfersOut → TraversingElement

4. Concepts

4.1. Component

A Component is a designed entity that exists to perform a Function. Example Components include launch vehicle, spacecraft, telecommunication subsystem, flight software, attitude control software, and mission operations team.
Supertypes
  • base:AggregatedElement
  • base:ContainedElement
  • base:Container
  • project:DevelopedElement
  • base:IdentifiedElement
  • PerformingElement
  • PresentingElement
  • project:RealizedElement
  • SpecifiedElement
  • project:SuppliedElement
Subtypes
  • fse:Assembly
  • fse:Subsystem
Relations with source
  • Component ← base:Aggregates → Component
  • Component ← base:Contains → Component
  • Component ← Induces → Environment
  • Component ← project:Realizes → Component
Relations with target
  • Component ← base:Aggregates → Component
  • Component ← base:Contains → Component
  • Mission ← Deploys → Component
  • Environment ← Influences → Component
  • Component ← project:Realizes → Component
  • project:WorkPackage ← project:Supplies → Component

4.2. Environment

An Environment represents a set of conditions under which a Component performs some Function. Example environments include low earth orbit, trans-Jupiter cruise, and Martian north polar surface.

Supertypes
  • base:AggregatedElement
  • base:IdentifiedElement
Relations with source
  • Environment ← base:Aggregates → Environment
  • Environment ← Influences → Component
Relations with target
  • Environment ← base:Aggregates → Environment
  • Component ← Induces → Environment

4.3. Flow

A Flow is a non-discrete transfer of substance or energy between Components. Non-discrete implies that the substance or energy has no natural boundaries. Examples of Flow include electric current, propellant, and heat.

Supertypes
  • base:ContainedElement
  • base:Container
  • base:IdentifiedElement
  • TraversingElement
Relations with source
  • Flow ← base:Contains → Flow
Relations with target
  • Flow ← base:Contains → Flow

4.4. Function

A Function is a purpose for which a PerformingElement is included in a designed system.

Supertypes
  • base:IdentifiedElement
  • project:RealizedElement
Relations with source
  • Function ← Invokes → Function
  • Function ← project:Realizes → Function
Relations with target
  • Function ← Invokes → Function
  • PerformingElement ← Performs → Function
  • Function ← project:Realizes → Function

4.5. Interface

An Interface represents a set of mechanical, electrical, signal, or other features that describe some aspect of a PresentingElement's connection to or interaction with another PresentingElement.

Supertypes
  • project:DevelopedElement
  • base:IdentifiedElement
  • PresentingElement
  • project:RealizedElement
Subtypes
  • fse:EndCircuit
Relations with source
  • Interface ← project:Realizes → Interface
  • Interface ← Transfers → TraversingElement
  • Interface ← TransfersIn → TraversingElement
  • Interface ← TransfersOut → TraversingElement
Relations with target
  • Junction ← Joins → Interface
  • Junction ← Joins1 → Interface
  • Junction ← Joins2 → Interface
  • PresentingElement ← Presents → Interface
  • Interface ← project:Realizes → Interface

4.6. Item

An Item is a discrete unit of material or information exchanged between two PresentingElements.

Supertypes
  • base:ContainedElement
  • base:Container
  • base:IdentifiedElement
  • TraversingElement
Subtypes
  • MaterialItem
  • Message
Relations with source
  • Item ← base:Contains → Item
Relations with target
  • Item ← base:Contains → Item

4.7. Junction

An Junction represents a set of mechanical, electrical, signal, or other properties that are common to a set of mutually connected Interfaces.

Supertypes
  • base:ContainedElement
  • base:Container
  • project:DevelopedElement
  • base:IdentifiedElement
  • project:RealizedElement
Subtypes
  • fse:Function
Relations with source
  • Junction ← base:Contains → Junction
  • Junction ← Joins → Interface
  • Junction ← Joins1 → Interface
  • Junction ← Joins2 → Interface
  • Junction ← project:Realizes → Junction
Relations with target
  • Junction ← base:Contains → Junction
  • Junction ← project:Realizes → Junction
  • TraversingElement ← Traverses → Junction

4.8. MaterialItem

A MaterialItem is a discrete unit of material exchanged between two Functions. Example MaterialItems include rock hammer and soil sample.

Supertypes
  • Item
Relations with source
  • MaterialItem ← base:Contains → MaterialItem
Relations with target
  • MaterialItem ← base:Contains → MaterialItem

4.9. Message

A Message is a discrete unit of information exchanged between two Functions. Example Messages include command, command response, and file contents.

Supertypes
  • Item
Relations with source
  • Message ← base:Contains → Message
Relations with target
  • Message ← base:Contains → Message

4.10. Mission

A Mission is a PerformingElement that pursues Objectives.

Supertypes
  • base:IdentifiedElement
  • PerformingElement
  • PresentingElement
  • project:RealizedElement
  • SpecifiedElement
  • project:SuppliedElement
Relations with source
  • Mission ← Deploys → Component
  • Mission ← Pursues → Objective
  • Mission ← project:Realizes → Mission
Relations with target
  • Mission ← project:Realizes → Mission
  • project:Project ← project:Supplies → Mission

4.11. Objective

An Objective represents a specific interest that one or more stakeholders have in the successful execution of a mission.

Supertypes
  • base:AggregatedElement
  • project:Concern
  • base:IdentifiedElement
Relations with source
  • Objective ← base:Aggregates → Objective
Relations with target
  • Objective ← base:Aggregates → Objective
  • Mission ← Pursues → Objective

4.12. Product

A Product is a tangible artifact that results from achievement of mission objectives.

Supertypes
  • base:AggregatedElement
  • base:IdentifiedElement
Relations with source
  • Product ← base:Aggregates → Product
Relations with target
  • Product ← base:Aggregates → Product
  • PerformingElement ← Produces → Product

4.13. Requirement

A Requirement specifies an assertion about a SpecifiedElement that must be true for every acceptable realization of that element.

Supertypes
  • analysis:CharacterizedElement
  • base:IdentifiedElement
Properties
  • hasRationale
Relations with source
  • Requirement ← Refines → Requirement
  • Requirement ← Specifies → SpecifiedElement
Relations with target
  • Requirement ← Refines → Requirement
